# TstPlatform 产品概念文档

> 核心理念：汽车电子自动化工程协作平台
> 基于与测试团队的需求沟通整理

---

## 一、问题定义

### 1.1 当前痛点

```
现状一：散装的自动化工程
├── 工程师A写了一套测试脚本 → 存在自己电脑上
├── 工程师B有类似需求 → 重新写一套
├── 工程师A离职 → 脚本丢失或无法维护
└── 结果：重复造轮子，能力无法沉淀

现状二：庞大的完整性工程
├── 中台提供"万能测试套件"
├── 包含所有可能的测试场景
├── 体积庞大，启动缓慢
├── 工程师只想验一两个点
└── 结果：望而却步，宁愿自己写

现状三：事业部与中台割裂
├── 中台不知道事业部的真实需求
├── 事业部的优秀脚本无法反哺
├── 各自为战，资源浪费
└── 结果：双输局面
```

### 1.2 根本原因

| 问题 | 根本原因 |
|------|----------|
| 脚本无法复用 | 缺乏共享平台和复用机制 |
| 庞大工程无用 | 缺乏按需裁剪的轻量级方案 |
| 事业部与中台割裂 | 缺乏开源协作的机制和平台 |

## 二、产品定位

### 2.1 一句话定义

**汽车电子自动化工程协作平台** —— 让测试工程师能够方便地共享、复用、积累自动化测试工程。

### 2.2 核心隐喻

```
GitHub for Test Automation

就像GitHub让代码协作成为可能，
TstPlatform让测试自动化工程协作成为可能。

不是管理"用例"，而是管理"工程"
不是"执行测试"，而是"复用工程"
```

### 2.3 与相关概念的区别

| 概念 | 传统理解 | TstPlatform |
|------|----------|-------------|
| 测试用例管理 | 管理测试步骤和预期结果 | 管理可执行的自动化工程 |
| 测试执行平台 | 远程执行测试脚本 | 本地复用+云端协作 |
| 测试资产库 | 标准用例集合 | 项目工程 + 功能模块双轨 |

## 三、核心概念

### 3.1 组织单元：项目 (Project)

```yaml
项目:
  name: 北汽ZCU
  type: 整车控制器

  modules:
    - 接入认证
    - 诊断服务
    - 网络管理
    - 刷写流程

  engineers:
    - 张三（负责人）
    - 李四
```

### 3.2 可复用单元：功能模块 (Module)

功能模块是跨项目的通用测试能力。

```yaml
功能模块:
  name: 接入认证测试
  category: 诊断通信

  # 来源：从多个项目工程中提炼
  sources:
    - 北汽ZCU/接入认证
    - VCU/接入认证
    - 其他项目贡献

  # 成熟度：基于使用情况和反馈
  maturity: 稳定版/测试版/草稿

  # 使用统计
  stats:
    downloads: 128
    projects_using: 15
    rating: 4.5
```

### 3.3 实体单元：工程 (Project)

工程是绑定到具体项目的自动化实现。

```yaml
工程:
  name: 北汽ZCU-接入认证测试
  project: 北汽ZCU
  module: 接入认证

  # 工具链
  toolchain: TSMaster  # 或 Vector

  # 质量状态
  quality: 已验证      # 已验证 / 开发中 / 草稿

  # 复用方式
  reusability:
    - 完整复用（一键验证）
    - 部分复用（选择子集）
    - 修改复用（Fork后修改）
```

## 四、用户旅程

### 4.1 场景一：首次使用

```
工程师小王，刚接到VCU项目，需要做接入认证测试

1. 打开TstPlatform
2. 创建新项目：VCU
3. 搜索"接入认证"
4. 发现：
   ├── 功能模块：接入认证（通用版，较复杂）
   └── 项目工程：北汽ZCU/接入认证（轻量，已验证）
5. 选择复用"北汽ZCU/接入认证"
6. 查看工程详情：
   ├── 工具链：TSMaster ✓（我有）
   ├── 复杂度：中等
   ├── 验证状态：已验证（5个项目使用）
   └── 修改建议：仅需修改请求ID
7. 一键下载工程
8. 本地修改（5分钟）
9. 本地验证通过
10. 上传回平台：VCU/接入认证
    └── 标记为"已验证"

结果：原本需要2天开发的测试，30分钟搞定
```

### 4.2 场景二：贡献与完善

```
工程师小张，使用北汽ZCU的接入认证工程发现一个问题

1. 本地修复bug
2. 验证通过后
3. 上传更新：
   ├── 版本说明：修复了XX场景下的超时问题
   └── 变更对比：显示修改内容
4. 平台通知原作者
5. 原作者审核通过，合并到主分支
6. 所有使用这个工程的项目收到更新提示

结果：一个问题修复，所有项目受益
```

### 4.3 场景三：功能模块演进

```
中台团队观察数据：

1. 发现"接入认证"功能在10个项目中使用
2. 但每个项目都有些微修改
3. 分析共性和差异
4. 提炼出更通用的"接入认证功能模块V2"
5. 支持参数化配置（请求ID、超时时间等）
6. 新项目可以直接使用V2，无需修改

结果：从项目经验中提炼标准能力
```

## 五、核心功能

### 5.1 工程管理

| 功能 | 说明 |
|------|------|
| 工程上传 | 支持TSMaster/Vector工程文件上传 |
| 工程验证 | 一键验证工程可执行性 |
| 版本管理 | Git版本控制，支持分支和合并 |
| 工程搜索 | 按项目、功能、工具链等多维度搜索 |

### 5.2 复用机制

| 功能 | 说明 |
|------|------|
| 完整复用 | 工程验证通过，可直接一键执行 |
| 部分复用 | 选择工程中的子测试集 |
| Fork修改 | 基于现有工程创建自己的版本 |
| 差异对比 | 对比不同版本或不同工程的差异 |

### 5.3 协作机制

| 功能 | 说明 |
|------|------|
| 贡献流程 | 提交修改 → 审核 → 合并 |
| 使用统计 | 下载量、使用项目数、评分 |
| 问题反馈 | 工程使用中的问题和建议 |
| 最佳实践 | 优秀工程的推荐和展示 |

### 5.4 功能模块库

| 功能 | 说明 |
|------|------|
| 模块提炼 | 从项目工程中提炼通用功能模块 |
| 模块演进 | 基于使用反馈持续优化模块 |
| 模块认证 | 中台认证的高质量保证模块 |
| 参数化配置 | 支持不同项目的个性化配置 |

## 六、设计原则

### 6.1 产品哲学

**"把麻烦留给自己，把简单留给用户"**

- 用户操作：搜索 → 选择 → 下载 → 使用（4步）
- 平台承担：版本管理、兼容性处理、质量评估、推荐算法

### 6.2 关键决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 管理单元 | 工程而非用例 | 工程师的实际工作单元是工程 |
| 复用粒度 | 支持多层级 | 完整/部分/Fork，满足不同场景 |
| 工具策略 | 工具无关 | Vector/TSMaster都是工具，不绑定 |
| 协作模式 | 开源共建 | 事业部和中台共同维护 |
| 演进路径 | 项目→模块 | 从实践中提炼，而非顶层设计 |

### 6.3 反模式警示

```
❌ 不要成为第二个Polarion
   - 使用成为负担
   - 线下搞搞更快
   → 确保平台真正提高效率

❌ 不要追求大而全
   - 万能测试套件
   - 启动慢，没人用
   → 支持轻量级复用

❌ 不要强加标准
   - 强制使用标准格式
   - 增加工程师负担
   → 兼容现有工作流，渐进改进
```

## 七、与TSMaster原型验证的关系

### 7.1 验证目标调整

原目标：验证TSMaster COM API可行性
新目标：验证"工程迁移到TSMaster后的可复用性"

### 7.2 试点用例选取原则

从"技术验证"转向"工程验证"：

| 原选取标准 | 新选取标准 |
|-----------|-----------|
| 技术覆盖度 | 工程代表性 |
| 服务完整性 | 复用价值高 |
| 复杂度分级 | 使用频率高 |

### 7.3 成功标准补充

原有技术指标 + 复用指标：
- [ ] TSMaster工程可以被其他项目复用
- [ ] 复用修改工作量 < 原开发工作量的30%
- [ ] 工程师愿意主动上传自己的工程

## 八、演进路线图

### Phase 1: 试点验证（3-6月）

```
目标：验证"工程复用"模式可行

关键动作：
├── 选择2-3个试点项目
├── 支持TSMaster工程上传/下载
├── 实现基础的工程搜索和复用
├── 3-5个核心功能模块的积累
└── 收集工程师反馈，验证使用频率

成功标准：
├── 试点项目工程师主动使用平台
├── 至少1个工程被跨项目复用
└── 复用节省时间 > 使用平台时间
```

### Phase 2: 功能完善（6-12月）

```
目标：完善平台功能，扩大覆盖

关键动作：
├── 支持Vector工程（存量迁移）
├── 实现版本管理和协作流程
├── 功能模块自动提炼
├── 扩展到更多事业部
└── 建立贡献激励机制
```

### Phase 3: 生态运营（12-18月）

```
目标：形成活跃的测试工程生态

关键动作：
├── 丰富的功能模块库
├── 中台与事业部良性互动
├── 优秀工程的标准化推广
├── 新人培训体系（基于平台）
└── 持续优化和迭代
```
