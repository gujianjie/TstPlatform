# 轻量级需求-工程匹配方案

> 放弃复杂结构化，用最小成本解决"是否符合"问题

---

## 一、问题重新审视

### 本质需求
使用者拿到一个测试工程，需要快速判断：**这玩意能不能用在我项目上？**

### 为什么结构化方案不对
- 前期工作量巨大（每个需求都要结构化）
- 使用者学习成本高（要学会R-Spec）
- 维护困难（需求变化要同步更新结构化数据）
- 冷启动问题（没有足够数据时匹配引擎无效）

---

## 二、轻量级方案：三层渐进式了解

### 第一层：工程名片（一眼判断）

用最简单的方式展示工程核心信息：

```
┌─────────────────────────────────────────┐
│ 工程：北汽ZCU-接入认证测试                 │
├─────────────────────────────────────────┤
│ 【一句话】测试ECU接入认证的完整流程         │
│                                          │
│ 【测了什么】                               │
│   ✓ 默认会话→扩展会话切换                  │
│   ✓ 安全解锁（Seed-Key算法）              │
│   ✓ 认证通过/失败判定                      │
│                                          │
│ 【适用场景】                               │
│   • 车型：乘用车ZCU                       │
│   • 阶段：开发验证、产线EOL                │
│   • 协议：UDS on CAN                      │
│                                          │
│ 【关键参数】可配置                         │
│   • 诊断请求ID（默认0x7E0）               │
│   • 安全级别（默认0x01）                  │
│                                          │
│ 【使用数据】128次下载 | 15个项目使用       │
└─────────────────────────────────────────┘
```

**核心设计原则**：
- 一句话说清价值
- 用 checklist 展示能力（直观）
- 标明关键参数（判断适配难度）
- 显示使用数据（社交证明）

### 第二层：执行预览（试跑判断）

与其让人读文档，不如直接看效果：

```
┌─────────────────────────────────────────┐
│ 预览模式（无需下载）                        │
├─────────────────────────────────────────┤
│                                          │
│ 【测试流程可视化】                          │
│                                          │
│  步骤1: 建立默认会话                        │
│    └── 发送: 10 01                         │
│    └── 期望: 50 01 00 32 01 F4            │
│    [▶ 执行] [查看详情]                     │
│                                          │
│  步骤2: 切换到扩展会话                      │
│    └── 发送: 10 03                         │
│    └── 期望: 50 03 ...                    │
│    [▶ 执行] [查看详情]                     │
│                                          │
│  步骤3: 请求种子                           │
│    └── 发送: 27 01                         │
│    └── 期望: 67 01 [Seed]                 │
│    [▶ 执行] [查看详情]                     │
│                                          │
│ [一键完整执行] [下载工程]                   │
│                                          │
└─────────────────────────────────────────┘
```

**核心机制**：
- 单步执行：可以一步一步试跑，看每一步做什么
- 完整执行：一键跑完看结果
- 边试边改：发现参数不匹配，当场修改测试

看"和我类似的项目"怎么用：

│    • 修改诊断ID                           │

│                                          │
└─────────────────────────────────────────┘

## 三、匹配判断的简化流程

### 用户决策路径

```
搜索"接入认证"
    │
    ▼
┌─────────────────┐
│ 浏览工程名片     │ ← 30秒判断"大概能不能用"
│ （第一层）       │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
能用      不确定
    │         │
    ▼         ▼
预览测试   查看更多
（第二层）  相似案例
    │      （第三层）
    ▼         │
验证可行    └────→ 确定能用
    │              │
    ▼              ▼
下载使用  ←──── 下载使用
    │
    ▼
本地修改（此时回馈机制启动）
```

### 关键交互设计

**搜索阶段**：
- 输入需求关键词（自然语言）
- 返回相关工程列表（按使用频率排序）
- 每个工程显示"名片"

**评估阶段**：
- 点击工程 → 进入预览模式
- 可以单步执行或完整执行
- 实时显示执行结果

**决策阶段**：
- 看到"类似项目"的使用方式
- 参考别人的修改点
- 判断自己的工作量

## 四、信息如何产生？（结合回馈机制）

### 信息来源矩阵

| 信息 | 来源 | 产生时机 |
|------|------|----------|
| 一句话描述 | 工程上传时必填 | 上传 |
| 测了什么（checklist） | 从工程代码自动提取 + 上传者补充 | 上传/执行 |
| 适用场景 | 上传者填写 + 使用者反馈 | 上传/使用 |
| 关键参数 | 从工程代码自动提取 | 自动 |
| 使用数据 | 平台统计 | 自动 |
| 执行流程 | 从工程代码解析 | 自动 |
| 相似项目案例 | 使用者回馈 | 使用回馈 |

### 渐进完善机制

```
工程上传
    │
    ├── 自动生成：基础信息（从代码解析）
    │   └── 执行流程、关键参数、checklist框架
    │
    ├── 上传者填写：名片信息
    │   └── 一句话描述、适用场景
    │
    └── 平台发布：V1.0（基础可用）
         │
         └── 使用者下载使用
              │
              ├── 使用者A：简单使用
              │   └── 回馈：使用报告（勾选使用场景）
              │
              ├── 使用者B：修改后使用
              │   └── 回馈：修改点记录 + 使用报告
              │       └── 平台生成"VCU使用案例"
              │
              └── 使用者C：咨询使用者B
                  └── 社区形成
```

### 回馈的具体形式（渐进引导）

**Level 1 - 使用确认（30秒）**
```
使用工程后弹出：
┌────────────────────────────┐
│ 这个工程在你的项目中能用吗？  │
├────────────────────────────┤
│ [直接可用] [修改后可用] [不适用]│
│                             │
│ 你的项目类型：[_下拉选择_]    │
│                             │
│ [提交]（帮助其他人找到合适的工程）│
└────────────────────────────┘
```

**Level 2 - 修改分享（2分钟）**
```
如果选择了"修改后可用"：
┌────────────────────────────┐
│ 分享你的修改（帮助类似项目）  │
├────────────────────────────┤
│ 你修改了什么：               │
│ □ 参数调整（ID、超时等）     │
│ □ 添加了测试步骤             │
│ □ 删除了不适用的步骤          │
│ □ 其他：_________           │
│                            │
│ 修改工作量：[_小/中/大_]     │
│                            │
│ 修改后的工程可以分享给其他人吗？│
│ [愿意分享] [项目敏感，不分享]  │
└────────────────────────────┘
```

**Level 3 - 深度贡献（10分钟+）**
```
- 上传修改后的工程（作为衍生工程或PR）
- 撰写使用心得/踩坑记录
- 帮助回答其他人的咨询
```

## 五、与回馈机制的衔接

### 闭环设计

```
使用者获取工程
    │
    ├── 查看名片（第一层）
    │   └── 信息来源：上传者 + 自动生成
    │
    ├── 预览测试（第二层）
    │   └── 信息来源：代码解析
    │
    └── 参考相似案例（第三层）
        └── 信息来源：历史使用回馈
                │
                ▼
        使用者决定使用
                │
                ▼
        本地修改（如果需要）
                │
                ▼
        触发回馈（渐进引导）
                │
                ├── Level 1: 使用确认
                │   └── 完善"使用数据"
                │
                ├── Level 2: 修改分享
                │   └── 完善"相似案例"
                │
                └── Level 3: 深度贡献
                    └── 完善工程本身
```

## 六、关键优势

| 维度 | 原方案（结构化） | 新方案（轻量级） |
|------|----------------|----------------|
| 前期工作量 | 高（每个需求都要结构化） | 低（从代码自动生成大部分信息） |
| 冷启动 | 困难（需要大量结构化数据） | 容易（上传即用，信息逐步完善） |
| 使用者门槛 | 高（要学R-Spec） | 低（浏览-预览-试用，直觉式） |
| 信息准确性 | 高（人工精确定义） | 中等（自动生成+社区完善） |
| 可持续性 | 低（维护成本高） | 高（社区驱动自我完善） |

## 七、待验证假设

| 假设 | 验证方式 |
|------|----------|
| 工程名片的信息足够做初步判断 | 找3-5个工程师测试 |
| 预览模式（单步执行）有实际价值 | 技术原型验证 |
| 使用报告回馈的质量足够 | 小规模试点 |
| 相似项目案例对决策有帮助 | 用户访谈 |

## 八、下一步建议

1. **设计工程名片模板**
   - 确定必须展示的信息字段
   - 设计视觉呈现方式

2. **验证信息自动生成**
   - 从TSMaster工程文件解析测试流程
   - 提取关键参数

3. **设计回馈表单**
   - Level 1（使用确认）：极简设计
   - Level 2（修改分享）：引导式设计

4. **小规模验证**
   - 选择2-3个现有Vector脚本
   - 手动创建工程名片
   - 找工程师测试判断流程
