## 1. 背景与痛点

### 1.1 企业测试现状

公司测试面临的三大核心问题：

| 问题 | 表现 | 影响 |
|------|------|------|
| **能力分散** | 多事业部测试工程师水平参差不齐，同类功能测试缺乏统一标准 | 重复造轮子，质量参差 |
| **自动化断层** | 中台Vector平台覆盖有限，事业部自行开发脚本无法回馈 | 平台方案难以推广，各干各的 |
| **成本高企** | Vector工具链License和设备成本高 | 资源受限，无法全面覆盖 |

### 1.2 现状架构示意

```
                    公司测试现状（多事业部）

    事业部A                    事业部B                    事业部C
    ┌─────────┐               ┌─────────┐               ┌─────────┐
    │ 资深工程师 │               │ 初级工程师 │               │ 中级工程师 │
    │ 自建脚本   │               │ 手工测试  │               │ 重复造轮子 │
    │ 不共享    │               │ 效率低   │               │ 质量参差 │
    └────┬────┘               └────┬────┘               └────┬────┘
         │                         │                         │
         └─────────────────────────┴─────────────────────────┘
                                   │
                    中台（Vector自动化平台）
                    ├── License成本高 → 覆盖度受限
                    ├── 需求响应慢 → 事业部等不及
                    └── 脚本闭源 → 事业部无法扩展

    结果：测试资产分散、重复投入、能力无法沉淀
```

## 2. 产品定位

### 2.1 核心目标

**不是**做一个替代Vector的工具，**而是**构建企业测试能力中台。

| 维度 | 定义 |
|------|------|
| **产品名称** | TstPlatform（汽车ECU自动化测试平台） |
| **目标用户** | 汽车电子工程师、测试工程师、产线测试人员 |
| **核心场景** | 汽车ECU的功能测试、诊断测试、刷写验证、产线EOL测试 |
| **产品形态** | 桌面应用（优先）+ 可选Web管理端 |
| **与TSMaster关系** | 上层应用平台 - 调用TSMaster COM API，提供更友好的测试管理体验 |

### 2.2 核心价值主张

1. **知识沉淀** - 测试用例标准化存储，支持Git版本控制
2. **能力拉齐** - 通过标准用例库和模板，提升整体测试水平
3. **渐进自动化** - 从手工到全自动的分级支持，降低自动化门槛
4. **成本优化** - 逐步从Vector迁移到TSMaster，降低License成本

## 3. 核心策略

### 3.1 三层解耦架构

```
┌─────────────────────────────────────────────────────────────────┐
│  第一层：测试知识沉淀（解决"能力不齐"问题）                         │
│  ├── 测试用例库（标准化、可复用）                                  │
│  ├── 测试方法模板（最佳实践固化）                                  │
│  └── 事业部贡献机制（优质用例反哺平台）                            │
├─────────────────────────────────────────────────────────────────┤
│  第二层：自动化引擎（解决"脚本迭代"问题）                          │
│  ├── 低代码/无代码编排（降低自动化门槛）                           │
│  ├── 开放插件机制（事业部可自主扩展）                              │
│  └── 与工具链解耦（支持Vector/TSMaster双栈）                     │
├─────────────────────────────────────────────────────────────────┤
│  第三层：工具链适配（解决"成本高"问题）                            │
│  ├── 优先TSMaster（降本）                                        │
│  ├── 兼容Vector（保护既有投资）                                    │
│  └── 统一抽象层（上层不感知底层差异）                              │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 工具链演进策略

| 阶段 | Vector | TSMaster | 说明 |
|------|--------|----------|------|
| 试点期 | 存量继续使用 | **新增主力** | 新用例优先TSMaster实现 |
| 推广期 | 逐步迁移 | 全面铺开 | 提供Vector→TSMaster迁移工具 |
| 成熟期 | 最小化维护 | **绝对主力** | 仅保留必要Vector License |

### 3.3 LLM能力应用

| 应用场景 | 功能描述 | 优先级 |
|---------|---------|--------|
| 用例描述规范化 | 非标准描述 → 标准YAML格式 | P1 |
| 用例生成辅助 | 基于需求文档自动生成用例框架 | P2 |
| 脚本转译 | Vector脚本 → TSMaster脚本 | P2 |
| 智能补全 | 用例编写时的智能提示 | P2 |

## 4. 技术架构

### 4.1 平台分层架构

```
┌────────────────────────────────────────────────────────────────────┐
│                         TstPlatform                                 │
├────────────────────────────────────────────────────────────────────┤
│  应用层                                                              │
│  ├── 用例库（标准用例模板、事业部共享库、版本管理-Git）                │
│  ├── 测试执行（手动/自动、单步调试、批量执行）                         │
│  ├── 结果管理（报告生成、趋势分析、缺陷关联）                          │
│  └── 知识沉淀（测试方法、最佳实践、培训材料）                          │
├────────────────────────────────────────────────────────────────────┤
│  引擎层                                                              │
│  ├── 用例解析引擎（YAML/JSON标准格式 → 可执行脚本）                   │
│  ├── 编排引擎（顺序、分支、循环、并行）                                │
│  ├── 判定引擎（PASS/FAIL/INCONCLUSIVE，支持自定义规则）               │
│  └── LLM适配层（用例转译、描述规范化、智能补全）                       │
├────────────────────────────────────────────────────────────────────┤
│  适配层（工具链抽象）                                                 │
│  ├── TSMaster适配器（主力，全功能支持）                               │
│  │      └── 基于COM API封装（参考Kimi报告实现）                       │
│  ├── Vector适配器（过渡，兼容存量）                                   │
│  │      └── CANoe/CANalyzer脚本桥接                                  │
│  └── 通用协议（原始CAN/LIN帧，不依赖特定工具）                         │
├────────────────────────────────────────────────────────────────────┤
│  硬件接口                                                            │
│  └── 同星TC系列 / Vector接口 / 其他（通过TSMaster硬件抽象层兼容）      │
└────────────────────────────────────────────────────────────────────┘
```

### 4.2 标准化用例格式

采用**分层描述**：上层统一，底层适配。

```yaml
# 标准用例格式示例：诊断会话控制测试
test_case:
  id: DIAG-SESSION-001
  name: 诊断会话切换测试
  category: 诊断通信
  level: L1-基础  # L0手工/L1半自动/L2脚本/L3全自动

  # ========== 标准化部分（所有事业部一致）==========
  standard:
    purpose: 验证ECU支持默认会话和扩展会话的切换
    preconditions:
      - ECU上电，总线通信正常
      - 诊断仪连接建立
    steps:
      - id: 1
        action: 发送诊断会话控制请求（默认会话0x01）
        expected: 收到肯定响应，进入默认会话
      - id: 2
        action: 发送诊断会话控制请求（扩展会话0x03）
        expected: 收到肯定响应，进入扩展会话
      - id: 3
        action: 读取当前会话状态（通过特定DID）
        expected: 返回扩展会话状态值

    pass_criteria: 所有步骤收到肯定响应，无NRC返回

  # ========== 适配部分（工具链相关）==========
  adaptation:
    tsmaster:
      script_ref: scripts/tsmaster/diag_session.py
      params:
        request_id: "0x7E0"
        response_id: "0x7E8"
        timeout_ms: 50

    vector:  # 存量迁移用
      script_ref: scripts/vector/diag_session.can
      params:
        # Vector特有参数
```

## 5. 演进路线图

### 5.1 三阶段规划

```
阶段        时间        目标                    关键动作
──────────────────────────────────────────────────────────────────────
试点期      3-6月      验证模式，建立标杆        选1-2个事业部，TSMaster为主
           │                              跑通用例库+自动化流程
           ▼
推广期      6-12月     扩大覆盖，完善平台        推广到更多事业部
           │                              Vector存量迁移工具
           ▼
成熟期      12-18月    生态运营，能力沉淀        LLM辅助用例生成/转译
```

### 5.2 试点期MVP功能（P0）

**必须包含**：

- [ ] 用例库管理
  - [ ] 标准用例模板定义
  - [ ] 用例CRUD（YAML编辑）
  - [ ] Git版本集成

- [ ] TSMaster适配器
  - [ ] COM API基础封装（参考Kimi报告实现）
  - [ ] 诊断服务调用（$10/$22/$2E等核心服务）
  - [ ] 报文收发与信号解析

- [ ] 测试执行
  - [ ] 单用例手动执行
  - [ ] 结果判定（PASS/FAIL）
  - [ ] 简单报告导出

- [ ] 试点支持
  - [ ] 1-2个事业部的试点接入
  - [ ] 5-10个标准用例的落地验证

**延后考虑（P1/P2）**：

- LLM转译功能（P1 - 试点验证后）
- Vector适配器完善（P1 - 存量迁移阶段）
- 多事业部协作功能（P2 - 推广期）
- 高级报告与数据分析（P2 - 成熟期）

## 6. 组织与职责

### 6.1 中台职责界定

| 职责类型 | 具体内容 | 边界（不做） |
|---------|---------|-------------|
| **平台运营** | 用例审核、质量管理、权限控制、版本发布 | 不干涉事业部具体项目排期 |
| **规范定义** | 用例标准格式、测试分级标准、评审Checklist | 不强制要求事业部必须用自动化 |
| **工具开发** | 平台核心功能、TSMaster适配器、迁移工具 | 不重复开发TSMaster已有功能 |
| **能力建设** | 培训材料、最佳实践案例、技术支持 | 不替事业部写具体项目用例 |

### 6.2 事业部参与机制

```
           事业部测试工程师
                  │
    ┌─────────────┼─────────────┐
    ↓             ↓             ↓
  提交优质用例   反馈改进建议    开发工具插件
    │             │             │
    └─────────────┴─────────────┘
                  ↓
           平台审核与标准化
                  ↓
    ┌─────────────┼─────────────┐
    ↓             ↓             ↓
  纳入标准库    积分/排名激励    工具能力增强
    │             │             │
    └─────────────┴─────────────┘
                  ↓
           全公司复用与改进
```

### 6.3 测试能力分级

| 阶段 | 事业部能力状态 | 平台支持方式 |
|------|---------------|-------------|
| L0 手工 | 只会手动操作 | 提供标准测试用例模板，指导执行 |
| L1 半自动 | 会单步执行 | 支持单步调试、结果记录 |
| L2 脚本辅助 | 会写简单脚本 | 提供脚本框架和常用函数库 |
| L3 全自动化 | 能独立开发 | 支持用例上传共享，纳入平台生态 |
