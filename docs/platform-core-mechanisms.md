# TstPlatform 核心机制设计

> 针对需求-测试关联、回馈增强机制的深度思考

## 一、问题一：需求与测试案例的关联管理

### 1.1 问题本质

```
现状：
需求文档 ──文字描述──→ 功能：接入认证
                           │
                           ▼
                   隐性知识（工程师脑中）
                           │
                           ▼
测试工程：北汽ZCU-接入认证测试

问题：
新工程师看到"接入认证"工程，不知道是否匹配自己的需求
因为需求文字描述 ≠ 测试工程能力范围（存在语义鸿沟）
```

### 1.2 解决方案：三层显性化机制

#### 第一层：需求结构化（R-Spec）

将文字需求转化为结构化测试需求：

```yaml
# 需求结构化模板
requirement_spec:
  id: REQ-ACCESS-AUTH-001
  name: 接入认证功能

  # 功能维度（What）
  function:
    category: 诊断通信
    standard: UDSonCAN
    services:
      - $10 会话控制
      - $27 安全访问

  # 测试维度（How to test）
  test_dimensions:
    - 正常流程：默认会话→扩展会话→安全解锁→认证通过
    - 异常流程：超时处理、错误密钥处理、会话保持
    - 边界条件：密钥尝试次数、会话超时时间

  # 适用场景（When/Where）
  applicability:
    ecu_types: [ZCU, VCU, BMS]
    project_stages: [开发验证, 产线EOL]

  # 关键参数
  key_parameters:
    - request_id: 诊断请求ID
    - response_id: 诊断响应ID
    - security_level: 安全级别
    - timeout_ms: 超时时间
```

#### 第二层：工程能力清单（Capability Manifest）

每个测试工程必须提供能力清单：

```yaml
# 工程能力清单
capability_manifest:
  project: 北汽ZCU
  module: 接入认证

  # 覆盖的测试维度
  coverage:
    - 维度：正常流程
      支持：完整支持
      详情：默认→扩展→解锁→认证
    - 维度：异常流程
      支持：部分支持
      详情：仅支持超时处理
    - 维度：边界条件
      支持：不支持

  # 关键参数配置
  configurable_params:
    - name: request_id
      type: hex
      default: "0x7E0"
      description: 诊断请求ID
    - name: timeout_ms
      type: int
      default: 50
      range: [10, 1000]

  # 工具链依赖
  toolchain: TSMaster
  hardware: [TC1016]

  # 质量信息
  quality:
    verified_in: [北汽ZCU, 另一项目]
    pass_rate: 98.5%
    execution_time: 30s
```

#### 第三层：匹配评估引擎

```
用户输入需求 → 结构化解析 → 与工程能力清单匹配 → 匹配度报告

匹配度报告示例：
┌─────────────────────────────────────────────────┐
│ 需求：VCU接入认证测试                              │
│                                                  │
│ 推荐工程：北汽ZCU-接入认证测试                      │
│ 整体匹配度：85%                                  │
├─────────────────────────────────────────────────┤
│ 匹配详情：                                        │
│ ✅ 正常流程（完全匹配）                            │
│ ✅ 诊断服务（完全匹配）                            │
│ ⚠️  异常流程（部分匹配：缺少错误密钥测试）          │
│ ❌ 边界条件（不匹配：需要添加密钥次数限制测试）      │
├─────────────────────────────────────────────────┤
│ 建议操作：                                        │
│ 1. Fork此工程                                     │
│ 2. 添加异常流程：错误密钥处理                       │
│ 3. 添加边界条件：密钥尝试次数限制                   │
│ 预计工作量：2-4小时                               │
└─────────────────────────────────────────────────┘
```

### 1.3 隐性知识显性化策略

| 隐性知识 | 显性化方式 | 载体 |
|---------|-----------|------|
| "这个工程测了什么" | 能力清单 | capability_manifest.yaml |
| "适用于什么场景" | 适用性矩阵 | applicability_matrix.md |
| "和需求的对应关系" | 需求-测试追溯 | traceability_matrix.md |
| "修改时要注意什么" | 修改指南 | modification_guide.md |

## 二、问题二：回馈增强机制

### 2.1 问题本质

```
平台工程 ──Fork──→ 本地工程 ──修改──→ 本地验证通过
    │                                      │
    │                                      │
    └──────── 如何回馈？ ←─────────────────┘

挑战：
1. 本地修改可能是项目特定的，不适合回馈
2. 工程师缺乏回馈动力
3. 回馈质量难以保证
4. 回馈与原始工程的关系管理
```

### 2.2 解决方案：四层回馈机制

#### 第一层：修改分类（Modification Taxonomy）

强制用户在修改时声明修改类型：

```yaml
modification_record:
  type: 项目适配 / 功能增强 / 缺陷修复 / 参数调整

  # 类型1：项目适配（通常不回馈）
  项目适配:
    example: 修改request_id从0x7E0到0x7A0
    feedback: 不需要
    reason: 项目特定配置

  # 类型2：功能增强（建议回馈）
  功能增强:
    example: 添加错误密钥处理流程
    feedback: 强烈推荐
    benefit: 提升工程完整性，其他项目受益

  # 类型3：缺陷修复（强烈建议回馈）
  缺陷修复:
    example: 修复超时判断逻辑错误
    feedback: 必须回馈
    severity: 高

  # 类型4：参数调整（选择性回馈）
  参数调整:
    example: 调整默认超时时间
    feedback: 可选
    condition: 如果调整具有普适性
```

#### 第二层：回馈通道（Feedback Channels）

```
┌────────────────────────────────────────────────────┐
│                    回馈通道体系                      │
├────────────────────────────────────────────────────┤
│                                                     │
│  通道1：Pull Request模式（针对功能增强/缺陷修复）      │
│  ───────────────────────────────────────────────  │
│  用户Fork工程 → 修改 → 提交PR → 原作者审核 → 合并    │
│                                                     │
│  通道2：经验报告模式（针对使用心得）                  │
│  ───────────────────────────────────────────────  │
│  用户使用 → 填写使用报告 → 平台审核 → 关联到工程     │
│                                                     │
│  通道3：参数建议模式（针对配置优化）                  │
│  ───────────────────────────────────────────────  │
│  系统收集参数调整 → 分析普适性 → 建议默认值变更      │
│                                                     │
│  通道4：衍生工程模式（针对项目特定适配）              │
│  ───────────────────────────────────────────────  │
│  项目适配修改 → 作为衍生工程上传 → 建立血缘关系      │
│                                                     │
└────────────────────────────────────────────────────┘
```

#### 第三层：激励机制（Incentive Mechanism）

| 激励维度 | 具体措施 |
|---------|---------|
| **声誉激励** | 贡献者排行榜、优质贡献者认证、专家标签 |
| **实用激励** | 贡献积分可兑换平台高级功能、优先技术支持 |
| **职业激励** | 贡献记录纳入绩效评估、晋升参考 |
| **社交激励** | 贡献者社区、线下交流活动、内部技术分享 |

**贡献等级**：
```
Level 1 - 使用者：下载使用工程
Level 2 - 反馈者：提交使用报告、问题反馈
Level 3 - 贡献者：提交PR并被合并
Level 4 - 维护者：成为工程维护者，审核他人PR
Level 5 - 专家：主导功能模块开发，指导他人
```

#### 第四层：质量控制（Quality Gate）

```
回馈内容 ──自动检查──→ 质量评分 ──人工审核──→ 平台收录
              │                      │
              ▼                      ▼
         ┌─────────┐           ┌─────────┐
         │自动检查项│           │审核要点 │
         ├─────────┤           ├─────────┤
         │功能完整性│           │回馈价值 │
         │代码规范 │           │通用性   │
         │可执行性 │           │文档完整性│
         │文档完整 │           │冲突检查 │
         └─────────┘           └─────────┘
```

### 2.3 回馈与原始工程的关系管理

#### 工程血缘图谱

```
北汽ZCU-接入认证（v1.0）【原始工程】
    │
    ├── PR#1: 添加错误密钥处理 ──合并──→ v1.1
    │   贡献者：张三
    │
    ├── PR#2: 优化超时判断逻辑 ──合并──→ v1.2
    │   贡献者：李四
    │
    └── Fork: VCU-接入认证【衍生工程】
        修改类型：项目适配
        主要变更：request_id修改
        │
        ├── PR#3: 添加密钥次数限制测试 ──合并到原始──→ v1.3
        │   贡献者：王五（VCU项目）
        │
        └── Fork: XCU-接入认证【衍生工程】
            修改类型：项目适配

# 血缘图谱价值：
# 1. 可以看到工程的演进历史
# 2. 可以发现跨项目的通用改进
# 3. 可以评估工程的影响范围
```

#### 回馈的自动汇聚

```
场景：多个衍生工程都做了类似的修改

北汽ZCU-接入认证（原始）
    │
    ├── 衍生1：VCU-接入认证
    │   └── 修改：添加密钥次数限制测试
    │
    ├── 衍生2：BMS-接入认证
    │   └── 修改：添加密钥次数限制测试
    │
    └── 衍生3：MCU-接入认证
        └── 修改：添加密钥次数限制测试

平台检测：3个衍生工程都添加了"密钥次数限制测试"

平台动作：
1. 自动识别这是一个通用需求
2. 向原始工程维护者发送建议
3. 建议将"密钥次数限制测试"合并到原始工程
4. 提供合并后的影响评估（会影响多少衍生工程）
```

## 三、两个问题的关联解决

### 3.1 闭环机制

```
┌─────────────────────────────────────────────────────────┐
│                      正向流程（需求→测试）                  │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  需求结构化 ──→ 工程搜索匹配 ──→ 获取工程 ──→ 本地使用      │
│       │                                        │         │
│       │                                        │         │
│       └──────── 反馈优化 ──────────────────────┘         │
│                                                          │
├─────────────────────────────────────────────────────────┤
│                      逆向流程（测试→需求）                  │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  本地修改 ──→ 修改分类 ──→ 选择回馈通道 ──→ 平台审核        │
│       │                                        │         │
│       │                                        │         │
│       └──────── 质量评估 ──────────────────────┘         │
│                                                          │
│  平台审核通过 ──→ 合并到原始/创建衍生 ──→ 更新能力清单      │
│                                                          │
│  能力清单更新 ──→ 改进匹配引擎 ──→ 更好地服务新需求         │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 3.2 关键设计原则

| 原则 | 说明 |
|------|------|
| **最小阻力原则** | 回馈流程不能超过5分钟，否则用户放弃 |
| **价值可见原则** | 让用户看到自己的回馈被使用和受益 |
| **渐进式原则** | 从简单回馈（使用报告）到复杂回馈（PR）逐步引导 |
| **社交化原则** | 让贡献成为一种社交货币，建立社区感 |
| **智能化原则** | 系统辅助判断回馈价值，降低人工审核成本 |
