### 文档更新概览

新增了3份核心文档：

#### 1. `docs/product-definition.md` - 产品定义文档
**核心定位**: TstPlatform 不是替代Vector的工具，而是**企业测试能力中台**

**三大痛点**:
- 能力分散: 多事业部测试水平参差不齐
- 自动化断层: Vector平台覆盖有限，事业部脚本无法回馈
- 成本高企: Vector工具链License和设备成本高

**三层解耦架构**:
```
第一层：测试知识沉淀（用例库、方法模板、贡献机制）
第二层：自动化引擎（低代码编排、开放插件、工具解耦）
第三层：工具链适配（TSMaster优先、Vector兼容、统一抽象）
```

**工具链演进策略**:
| 阶段 | Vector | TSMaster |
|------|--------|----------|
| 试点期 | 存量继续 | 新增主力 |
| 推广期 | 逐步迁移 | 全面铺开 |
| 成熟期 | 最小化维护 | 绝对主力 |

**MVP功能（P0）**:
- 用例库管理（YAML格式、Git版本集成）
- TSMaster适配器（COM API封装、诊断服务$10/$22/$2E）

## 2026-03-13 会话记录（续）

### 与测试团队的需求沟通

#### 核心洞察

测试团队提出了一个非常清晰的平台愿景：**以"项目-功能模块"为核心的自动化工程复用平台**

#### 关键概念

**1. 组织方式：项目 + 功能模块**
```
北汽ZCU项目
├── 接入认证（功能模块） ← 自动化工程
├── 诊断服务（功能模块） ← 自动化工程
└── 网络管理（功能模块） ← 自动化工程

VCU项目
├── 接入认证 ← 直接复用北汽ZCU的接入认证工程，小改即用
├── 刷写流程 ← 复用通用刷写模块
└── DTC管理 ← 新建或复用
```

**2. 工作流程**
```
工程师本地开发调试自动化工程
           ↓
    工程验证通过，质量有保证
           ↓
    上传平台（支持一键验证或快速验证）
           ↓
    其他工程师发现类似需求
           ↓
    选择复用：直接复用 / 基于现有工程修改 / 新建
           ↓
    持续积累 → 形成功能模块库（跨项目的通用测试能力）
```

**3. 复用层级**
| 层级 | 说明 | 示例 |
|------|------|------|
| 项目工程 | 具体项目的完整自动化 | 北汽ZCU接入认证测试工程 |
| 功能模块 | 跨项目的通用功能 | 接入认证功能模块（综合多个项目经验） |
| 轻量级 | 只验一两个点的精简版 | 从项目工程中裁剪的精简验证 |

**4. 解决的核心问题**

| 问题 | 解决方案 |
|------|----------|
| 脚本工程如何传递、复用、积累 | 平台化存储，基于项目的工程库 |
| 相同/类似功能如何传递给其他工程师 | 功能模块库 + 项目工程库双轨 |
| 自动化工程如何维护 | 开源协作，中台+事业部共建 |
| 事业部与中台壁垒 | 需求透明，双向受益，开源共享 |
| 庞大工程没人愿意用 | 支持轻量级复用，按需裁剪 |

#### 平台核心要求

**1. 好用（最基本）**
- 真正提高效率，不能成为工作累赘
- 反例：Polarion（使用成为负担）
- 目标：线下搞搞更快 → 用平台更快

**2. 定制化能力在公司自己手里**
- 需求开发及升级完善自主可控
- 有利于积累和与时俱进
- 平台的灵活性

**3. 工具无关**
- Vector还是同星，都只是工具
- 可以上传Vector的，也可以上传同星的
- Base从中台来，不绑定特定工具

**4. 把麻烦留给自己，把简单留给用户**
- 开发工具的基本理念
- 公司目前缺乏这种产品思维

#### 对平台设计的启示

**关键设计原则**：
1. **项目导向** - 不是用例库，是项目工程库
2. **功能模块化** - 可复用的最小单元是功能模块
3. **渐进式积累** - 从项目工程到功能模块的自然演进
4. **开源协作** - 事业部和中台共建共享
5. **轻量级优先** - 支持快速复用，避免大而全

**产品形态调整**：
- 不是"测试用例管理平台"
- 而是"汽车电子自动化工程协作平台"
- 核心是：工程的上传、验证、复用、积累

#### 与之前规划的关联

原有的TSMaster原型验证计划仍然有效，但视角需要调整：
- 不只是验证技术可行性
- 更要验证"工程迁移到TSMaster后的可复用性"
- 试点用例应该来自真实的项目工程
